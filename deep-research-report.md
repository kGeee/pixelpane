## Users and core jobs

The first release should target three user groups with clear, repeated needs. The first is students and self-learners who encounter dense textbook passages, lecture slides, diagrams, and problem statements that they do not fully understand. The second is multilingual professionals who need to move quickly through messages, emails, internal docs, and support threads without breaking flow. The third is everyday users and travelers who want to translate menus, product pages, PDFs, signs, and webpages without opening a separate translation workflow. Those are all high-frequency situations where “see text → ask for help” is natural. citeturn11view1turn11view2turn9view1turn9view2

The educational segment is especially attractive because the best learning-focused AI products increasingly emphasize guided understanding rather than answer dumping. OpenAI’s Study Mode is explicitly designed around step-by-step guidance, calibration to the learner’s level, and support for deeper understanding, while entity["organization","Khan Academy","education nonprofit"] positions Khanmigo as a tutor that walks learners through answers instead of simply returning them. That is exactly the right model for the product’s “Explain” experience when the source material looks academic. citeturn11view1turn11view2

The core jobs to be done are straightforward:

- **When I see a message in another language, translate it instantly and help me understand tone and meaning.**
- **When I hit a dense passage in a textbook or PDF, explain it in plain English at my level.**
- **When I open a restaurant menu or brochure PDF, translate the visible page and clarify unfamiliar terms.**
- **When I capture an error message, technical text, or a form, tell me what it means and what to do next.**

Those jobs are all compatible with a screenshot-first product because the user does not need the underlying app to expose text selection APIs. They only need to be able to see the content on screen. ## Product requirements

The product should have one defining loop: **invoke, select, understand, continue**. A hotkey opens the overlay. The user drags a region or chooses a window. OCR runs locally. An action rail appears with the primary choices. The result appears in a small floating panel next to the selection, with room for one follow-up question. The panel should stay lightweight and contextual; the user should not feel like they have been thrown into a full chat app just to translate a sentence. fileciteturn0file0 citeturn1search1turn1search2turn6search1

The launch version should keep the top-level actions opinionated and minimal. The top row should be **Translate**, **Explain**, **Simplify**, **Extract Text**, and **Ask**. **Summarize** should exist, but it should not be a primary button because it is too generic and does not differentiate the product. **Debug** should be contextual rather than always visible; it should appear when the OCR looks like code, logs, or an error message. That keeps the experience clean for general users while still serving technical users well. fileciteturn0file0

The result format should change based on the content, even if the UI stays the same. A **Message** output should show the translation first, then a short note on tone or implied meaning, then optional reply suggestions. A **Study** output should define key terms, simplify the passage, and optionally ask one clarifying question before going deeper. A **Menu** output should preserve the original dish names, translate the description, and add short notes for terms that a literal translation would not make obvious. A **Technical** output should explain the issue and suggest the next step. This is how the product feels purpose-built rather than generic, and it is fully achievable with lightweight content classification and prompt routing. citeturn11view1turn11view2turn12search0

For PDFs, the v1 requirement should be practical rather than overambitious. The product should support translating a visible page or a selected region anywhere on screen, including an online restaurant menu or brochure opened in a browser or PDF viewer. It can also support direct PDF import for single-page or page-by-page translation once the capture loop is stable. Full multi-page layout-preserving export is useful, but it should be a later addition, not the headline for launch. The category winners all succeed by eliminating context switching, and the visible-page flow already solves the user’s immediate problem. citeturn9view1turn0search1turn8view2

There are also some deliberate non-goals. The product should not launch with autonomous app control, browser automation, persistent personal memory, continuous recording, or a saved screen timeline. The right v1 promise is explicit, user-triggered understanding of a selected region. That keeps scope realistic, aligns with the original feasibility baseline, and preserves trust. fileciteturn0file0 citeturn8view2turn11view3

## Privacy and quality rules

Privacy needs to be central to the product narrative because macOS screen capture is permission-sensitive. Apple’s user guidance makes clear that users can decide which apps are allowed to record the screen, and Apple explicitly recommends that users evaluate an app’s privacy practices before granting access. Onboarding therefore needs a concrete promise, not a vague reassurance: the app only captures what the user explicitly selects, never records continuously, deletes temporary screenshots by default, and only sends data off device when the selected mode requires it. citeturn11view4

Local OCR should be mandatory in every plan. A local-only mode for translation and explanation should also exist, because it is valuable for privacy-sensitive users and companies. But it should be offered as a product choice, not imposed as the default if it degrades quality. Apple notes that offline or on-device translation can be less accurate than server-processed translation, which is exactly how the tradeoff should be messaged here: **local mode for privacy, cloud mode for best quality**. citeturn9view2

The product should also surface uncertainty instead of hiding it. Every answer should make it easy to see the detected language, whether the text came from OCR or from a structured PDF import, and whether a phrase is ambiguous or culturally specific. This matters because translation failures are often contextual rather than grammatical. A product that signals uncertainty clearly will earn more trust in study, work, and travel scenarios than one that pretends every result is definitive. citeturn12search0

**Launch release** should ship the universal selection loop: global shortcut, dimmed overlay, local OCR, language detection, the five primary actions, floating result panel, follow-up input, copy/export controls, ephemeral capture handling, and local-vs-cloud privacy settings.
